## Major Progress This Week

The week was dominated by identity infrastructure and messaging polish, with three distinct feature areas landing:

**Username System (2026-03-22 late)** ‚Äî Every account now requires a unique username. New users choose one at signup; existing accounts are intercepted at login and routed to a migration screen. Uniqueness is enforced via an atomic Firestore transaction writing both `users/{uid}` and `usernames/{usernameLower}` simultaneously. Signup flow was restructured: SignupScreen now handles only Auth account creation; the Firestore profile write was moved to VerifyEmailScreen to avoid storing unverified accounts.

**Run Level Phase 1 (2026-03-22)** ‚Äî Runs now carry a creator-set competitiveness tag (`casual | mixed | competitive`). A 3-pill picker in the Start-a-Run modal, color-coded badges on run cards (Casual = green, Competitive = red, Mixed = hidden as neutral default), and a filter in the ViewRunsScreen bottom sheet. Fully backwards-compatible ‚Äî old runs without the field are treated as `mixed`. No backend deploy required.

**Run Chat completion (2026-03-22)** ‚Äî Two missing pieces wired: (1) Chat expiry ‚Äî each run's chat closes 4 hours after `startTime`; expired chats show a read-only banner and are hidden from the Messages inbox. (2) Unread detection ‚Äî `sendRunMessage` stamps `lastMessageAt` on the run doc; `markRunChatSeen` writes `lastReadAt` on the participant doc; `useMyRunChats` derives `isUnread` and exposes `runChatUnreadCount`. HomeScreen Messages badge now shows total unread = DM unreads + Run Chat unreads.

**EditProfileScreen (2026-03-22)** ‚Äî New screen in ProfileStack for editing Display Name and Skill Level. Username shown read-only (not changeable after claim). Accessible via Settings ‚Üí Account Info.

Earlier in the week (2026-03-21), the DM system and Messages inbox landed: `dmService.js`, `MessagesScreen` (unified inbox with DMs + Run Chats), `DMConversationScreen`, and `OnboardingRegionScreen` (new step 0 of onboarding explaining Austin TX geographic focus).

## Technical Debt

- **`notifCooldowns` map grows unboundedly** ‚Äî Phase 1 notification functions store dedup keys as a flat map on `users/{uid}.notifCooldowns`. Will approach Firestore's 1 MB doc limit for power users (~500+ active users). Migration plan: move to `users/{uid}/notifCooldowns/{key}` subcollection with TTL policy. Not urgent for launch (BACKEND Known Issue #9).
- **`sendDMMessage` not transactional** ‚Äî two sequential writes (message subcollection + conversation metadata). Under a tight race condition, `lastMessage` on the conversation doc can lag by one message. Acceptable at MVP scale.
- **`useMyRunChats` does N `getDoc` calls** ‚Äî one per active run to fetch gymName + startTime. Fine for 0‚Äì3 runs; causes read spikes for power users. Acceptable for MVP.
- **`useTaggedClips` was recently improved** ‚Äî switched from 100-doc client-side filter to native `taggedUserIds array-contains` query (2026-03-19). Composite index required ‚Äî auto-create link appears in dev console on first run.
- **`constants/theme 2.js`** ‚Äî possible stale duplicate of `constants/theme.js`. Should be audited and deleted if no imports exist. Low risk to defer.
- **SignupScreen JSDoc comment** ‚Äî still describes the old flow (Firestore write in signup). Code is correct; comment is stale. Minor.
- **Username retry edge case** ‚Äî if SignupScreen Auth creation succeeds but the user closes the app before VerifyEmailScreen runs, they'll have an Auth account with no Firestore profile. On next login they hit ClaimUsername which handles it gracefully via `set({ merge: true })`, but they'll need to re-enter name/age/skill level via EditProfileScreen. Acceptable for MVP.
- **Gym images still on external hosts** ‚Äî 5 of 8 gyms use third-party image URLs. Only Fitness Connection migrated to Firebase Storage. Seed script warns on each. Migrate when convenient.
