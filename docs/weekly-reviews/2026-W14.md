## Technical Debt

- **`notifCooldowns` map grows unboundedly** — Flat map on `users/{uid}.notifCooldowns`; approaches Firestore's 1 MB doc limit for power users at ~500+ active users. Migration plan: move to `users/{uid}/notifCooldowns/{key}` subcollection with TTL policy. Not urgent at launch scale.
- **`sendDMMessage` not transactional** — Two sequential writes (message subcollection + conversation metadata). Under tight race, `lastMessage` on the conversation doc can lag. Acceptable at MVP scale.
- **`useMyRunChats` does N `getDoc` calls** — One per active run. Fine for 0–3 runs; causes read spikes for power users with many active runs.
- **`constants/theme 2.js`** — Possible stale duplicate of `constants/theme.js`. Audit and delete if no imports exist. Low risk.
- **`SignupScreen.js` JSDoc comment** — Still describes the old Firestore-write-in-signup flow. Code is correct; comment is stale.
- **Username retry edge case** — If SignupScreen Auth creation succeeds but the user closes before VerifyEmailScreen runs, they have an Auth account with no Firestore profile. `ClaimUsernameScreen` handles it via `set({ merge: true })` but they must re-enter name/age/skillLevel via EditProfileScreen.
- **Gym images still on external hosts** — 5 of 9 original gyms use third-party image URLs; 10 Batch 2 gyms have no images yet. Migrate when convenient.
- **`totalLateCancelled` field in backend** — `reliabilityService.calculateReliabilityScore` uses `totalLateCancelled` in its formula but it is not confirmed written by backend Cloud Functions (only `totalCancelled` is documented). Display fallback may be slightly inaccurate. Low severity.
- **Duplicate notification for gym-follower + scheduled-visitor overlap** — A user who both follows a gym AND has a scheduled visit will receive two run-creation notifications. Acceptable for V1 (small user base, rare overlap).
